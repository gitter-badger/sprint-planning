Sprint Planning

The agile methodology «Scrum» uses a developer planning meeting to identify and commit to the work which will be completed in each sprint. The aims of the meeting are [1]:

  1. Select what work is to be done
  2. Prepare the Sprint Backlog which details the work and the time which will be taken
  3. Identify and communicate how much of the work is likely to be done during the current sprint

Each of these aims requires estimation. With scrum the work is assigned to the whole team. The whole team should discuss the work, agree to the estimation, and collectively commit to a set of demonstrable features or fixes within the next sprint.


One challenge of group estimation for work is the cognitive bias of anchoring. The first person to state an estimate influences the people who subsequently provide an estimate. This creates a skewed consensus which is influenced by the order in which people speak.


An established approach to counter anchoring is to have each developer place an estimate face down onto the table. A standard deck of 52 playing cards can be used for this purpose. Once all of the team have placed a hidden estimate on the table they may all be revealed at once and discussed. Multiple rounds of estimation and discussion can then be undertaken to form the team consensus. This planning and estimation activity which is known as «Poker».


This site provides electronic version of cards so that the team does not have to be in the same room to make estimates.

Frequently Asked Questions


What equipment do I need to use this site?


You need individual screens and audio communications to the people you are working with. This could be achieved by all sitting in a meeting room with a set of smart phones or tablets. Alternatively individuals may sit at remote locations and use a teleconferencing service.


What information do I need to provide to use this site?


None. The service is designed to be anonymous. You can use a feature number or another anonymous identifier as your planning room name. It is recommended that you do not use any company names, product names, feature names, bug names, employee names or any other identifiable information. The site does not use cookies although it may use an analytics service at a future date to provide information to improve the service.


What data is retained by this site?


The application server logs the room names. It is recommend that you use a distinct but unidentifiable room name such as an anonymous feature number as your planning room name. If you are using a teleconference service you could use the meeting number of pin as the room name.


Where do I submit bugs and fixes?


Raise an issue over at the GitHub project page or better yet fork the code, fix it, and send us a pull request.
